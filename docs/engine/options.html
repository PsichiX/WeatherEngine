WeatherEngine - Game-ready weather simulations

Options and keeping it stable

create_sim takes a SimOptions struct - the solver’s tuning knobs. You get a validated set from default_sim_options() and override the few you need. Most fields you can leave alone. This page explains the ones that matter, with a focus on the one thing that trips people up: keeping the simulation stable so a bad number does not blow up (or, here, come back as an Error).

Start every time from the defaults:

SimOptions options = default_sim_options();
// ... override a couple of fields ...
client->create_sim(config, axis, std::move(sounding), options);

Why stability is a thing at all

The engine advances the weather in small time steps. There is a hard limit on how big each step can be relative to the cell size: if a step is too big for the cells, the numbers explode. This is a universal fact of grid-based physics (the “CFL condition” - think of it as “a wave must not jump more than one cell per step”). WeatherEngine checks this for you and, if you violate it, returns an Error event instead of running an unstable sim. So stability tuning here is really about staying under that limit while stepping as fast as you can.

There are two waves to worry about: the slow weather itself, and fast sound waves. Sound is the awkward one, because it moves far faster than any storm, which would force absurdly tiny steps. The engine uses a trick for it.

The time step: dt_per_meter

The base time step is not set directly; it scales with your finest cell size:

  • dt_per_meter - seconds of time step per meter of finest cell, s/m. The actual dt is roughly dt_per_meter * dx_finest. The default is tuned so the weather CFL is satisfied at normal cell sizes. Bigger = faster but closer to the edge of stability; smaller = safer but slower.

You rarely change this. The SimReady status reports the resulting dt in seconds so you can see what you got.

The sound-wave trick: sound_speed and acoustic_substeps

Real sound travels at ~340 m/s. Resolving it directly would force a tiny time step. The engine instead uses a slower artificial sound speed and takes several cheap acoustic substeps within each weather step, which keeps the pressure field consistent without crippling the step size.

  • sound_speed - the artificial sound speed, m/s. Lower means a bigger stable step (cheaper), but it must stay above the fastest air motion in your storm or the pressure solve misbehaves. The examples use 120 m/s - comfortably above storm updrafts (tens of m/s) yet far below real sound.
  • acoustic_substeps - how many acoustic substeps per weather step. More substeps let you use a lower sound_speed or larger cells while staying stable. The examples use 7.
  • off_centering - a small forward-weighting (~0.2) that damps acoustic noise. Leave at the default unless you see pressure ringing.

The one override you will almost always make

The default sound_speed/acoustic_substeps are tuned for a particular cell size. At the example’s 200 m cells the acoustic CFL is a touch tight, so the examples loosen it exactly like this:

SimOptions options = default_sim_options();
options.sound_speed = 120.0f;      // slower artificial sound
options.acoustic_substeps = 7;     // enough substeps to stay stable

If create_sim returns an Error mentioning an acoustic CFL violation, this pair is the fix: lower sound_speed and/or raise acoustic_substeps until it is accepted. (Internally the acoustic CFL must stay at or below 0.35; you do not compute it - you just nudge these two until the config is accepted.)

Mixing and reference density

  • diffusivity (m²/s) - how strongly the solver smooths small-scale noise, a stand-in for turbulent mixing. Higher = smoother, more diffuse storms; lower = sharper but noisier. Default is a good balance.
  • air_density (kg/m³) - a reference air density (~1.2 at sea level). Leave at default unless you are matching a specific setup.

Advection: how stuff is carried by the wind

advection picks how the solver moves quantities along with the flow (the interpolation used when tracing where each cell’s air came from):

AdvectionSchemeCharacterCost
Linearcheap, but smears/diffuses sharp featureslowest
VelocityCubicsharp winds, smooth scalars - the defaultmedium
MonotoneCubicsharpest everywhere, no overshoothighest

Stick with the default VelocityCubic unless you have a reason. Drop to Linear to save time when you do not care about crisp cloud edges; go to MonotoneCubic for the cleanest visuals at more cost.

  • skip_quiescent - a performance option: copy undisturbed (“quiet”) bricks straight through instead of advecting them. Safe speedup for domains with lots of calm air; leave it on if the default enables it.

The boundaries: sponges and lids

Your domain has edges, and weather hitting an edge would reflect back inward and corrupt the solution - like waves bouncing off the walls of a bathtub. Sponge layers are soft zones near the edges that gently absorb outgoing disturbances so they leave cleanly.

  • side_sponge - width, in base cells, of the absorbing zone on the x and y sides (a VoxelDims; 0 on an axis disables it there). Keep some side sponge if storms might approach the horizontal edges.
  • top_sponge - width, in cells, of the absorbing zone at the domain top. Important: a storm’s updraft slams into the top, and without a top sponge (or a lid) that energy reflects down. 0 disables it.
  • sponge_strength - how hard the outermost sponge layer damps. Too weak leaks reflections; too strong can reflect off the sponge itself. Default is balanced.
  • rigid_lid - instead of (or with) a top sponge, treat the domain top as a solid ceiling (zero vertical wind there each substep). Realistic when your top sits in the stable stratosphere, which naturally caps storms. Combine a modest top_sponge with the storm top clearing the EL (from Building a world) for clean tops.

If storm tops look like they are bouncing or piling unnaturally, revisit top_sponge, rigid_lid, and whether your domain is tall enough.

Parallelism

  • parallel - fan each solver pass out over the worker’s thread pool. This is a pure speed switch: the result is bitwise-identical whether on or off, so turning it on never changes the weather, only how fast it steps. Leave it on for real runs. (Recall the pool size is capped by new_sim_client, from How the engine is shaped.)

A sensible starting SimOptions

For a 200 m domain, this is all most projects touch:

SimOptions options = default_sim_options();
options.sound_speed = 120.0f;     // loosen acoustic CFL for 200 m cells
options.acoustic_substeps = 7;
// everything else: keep the validated defaults.

Then, only if you hit a problem:

  • Error about acoustic CFL -> lower sound_speed, raise acoustic_substeps.
  • Storm tops reflecting -> raise top_sponge or set rigid_lid, check domain height vs. EL.
  • Storms too noisy -> raise diffusivity or use MonotoneCubic.
  • Too slow -> ensure parallel is on, consider Linear advection and skip_quiescent, or coarsen the grid.

Takeaways

  • Always start from default_sim_options() and override sparingly.
  • Grid physics has a hard stability limit; break it and create_sim returns an Error, not a crash. The usual fix for a 200 m domain is sound_speed = 120, acoustic_substeps = 7.
  • sound_speed / acoustic_substeps trade step size against cost via the artificial-sound trick; dt_per_meter scales the base step with cell size.
  • advection picks sharpness vs. cost (default VelocityCubic); diffusivity smooths noise.
  • Sponges (side_sponge, top_sponge, sponge_strength) and rigid_lid stop weather reflecting off the domain edges - mind the top.
  • parallel is a free, bitwise-identical speedup.
