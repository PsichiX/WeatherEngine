WeatherEngine - Game-ready weather simulations

Getting data back out

A running simulation is useless until you can see it. This page covers every way to read the weather back: grabs (dense 3D volumes for the renderer), probes (a full weather report for one column), and queries (cheap scalars and the precipitation map). All of them come back as events you drain from poll().

Pick the lightest tool that answers your question. A grab copies a whole volume every frame; a query copies a handful of numbers. Do not grab when a query will do.

Grabs: dense voxel volumes for rendering

A grab samples the simulation onto a regular 3D voxel box and hands you the raw floats, ready to upload as a 3D texture. This is how you render clouds and rain. Two flavors:

  • grab_presets - render-ready derived channels. Prefer this for visuals.
  • grab_fields - the raw prognostic fields (Theta, Vapor, W, …). Use for analysis, debug views, or custom shading.

The ViewBox: what window to sample

Both grabs take a ViewBox describing the horizontal window and the output resolution:

ViewBox view{};
view.min = Vec2{-51200.0, -51200.0};      // low corner, world meters
view.max = Vec2{ 51200.0,  51200.0};      // high corner, world meters
view.resolution = VoxelDims{128, 128, 48};  // output voxels x, y, z
  • min / max are the horizontal world box to sample (they need not match the whole domain - zoom in on the storm if you like).
  • resolution is the output voxel count. The z voxels are uniform slices over the full domain height, regardless of the stretched simulation axis
    • so the grab is a plain box you can drop straight into a 3D texture.

Note the grab resolution is independent of the simulation grid. You can render a 128x128x48 texture from a domain with millions of fine cells, or sample a small area at high resolution. Build the ViewBox once and reuse it every frame.

Receiving a grab and the recycle loop

A grab answers with a Grabbed event carrying a ViewGrab buffer. You move it out with take_grab(), read its channels, then recycle the spent buffer so the next grab of the same shape reuses the allocation:

case SimEventKind::Grabbed: {
    Box<ViewGrab> grab = event->take_grab();

    VoxelDims d = grab->dims();                 // voxel counts
    Vec3 origin = grab->origin();               // world corner of the volume
    Vec3 vsize  = grab->voxel_size();           // world size of one voxel
    double when = grab->grab_time();            // sim time it was sampled

    for (uint32_t c = 0; c < grab->channel_count(); ++c) {
        Slice<const float> ch = grab->channel(c);  // flat volume, nx*ny*nz
        // ... upload ch.data(), ch.size() as a 3D texture slice ...
    }

    client->recycle(std::move(grab));   // hand the buffer back - no per-frame alloc
    break;
}

Two important details:

  • Channel layout is z-slice major: the flat array is nx*ny*nz floats, with x varying fastest, then y, then z. Index (x,y,z) is at x + y*nx + z*nx*ny.
  • Always recycle the buffer after uploading. The worker keeps a free list of buffers; recycling one lets the next same-shaped grab reuse it instead of allocating. In a per-frame loop this is the difference between zero and constant allocation churn.

Metadata to place the volume in the world

origin() and voxel_size() tell your renderer exactly where the volume sits and how big each voxel is in world meters, so you can position the 3D texture correctly. grab_time() is the simulation time (seconds) the grab was sampled - useful because the grab you receive this frame may be from a step or two ago (the worker runs ahead of your polling).

Probes: a full report for one column

A probe lifts a parcel from one world position and returns the same SoundingReportData you met in CAPE and shear - but for the live atmosphere, right now, at that spot:

client->probe(Vec2{0.0, 0.0});
// ... later, draining events ...
case SimEventKind::Probed: {
    SoundingReportData r = event->probe_report();
    if (r.valid) {
        std::printf("live CAPE %.0f, CIN %.0f, LCL %.0f m\n",
                    r.cape, r.cin, r.lcl);
    }
    break;
}

This is the tool for gameplay and telemetry that needs meteorological meaning rather than raw fields: show the player a “storm strength” readout, decide when conditions are tornado-favorable, trigger an event when live CAPE at a location crosses a threshold. valid is false if the position fell outside the domain. Because the storm modifies its own environment, a live probe drifts from the original sounding as the run goes on - that is expected and useful.

Queries: cheap scalars and the rain map

When you want a single number, not a volume, use a query. These are the cheapest reads.

query - a status snapshot

Returns a Status event with SimStatusData: current sim time, dt, step_count, domain size, domain_top, dx_finest. Use it to keep a HUD in sync without advancing.

query_max - domain-wide peak of fields

Returns a Values event with one value per requested field, in order - the maximum |value| of each over the whole domain:

Field fields[] = {Field::W, Field::Cloud, Field::Rain};
client->query_max(Slice<const Field>{fields, 3});
// ... draining ...
case SimEventKind::Values: {
    Slice<const float> v = event->values();
    std::printf("peak updraft %.1f m/s, peak cloud %.2e kg/kg\n", v[0], v[1]);
    break;
}

Perfect for “how strong is the storm right now” checks - peak updraft (W), peak condensate - without copying any volume. For query_max, the shape value_dims() is {n, 1, 1}.

query_precipitation - the accumulated rain map

Returns a Values event holding the surface precipitation map in millimeters - how much rain has accumulated at each base grid cell since the sim began. Here value_dims() is {nx, ny, 1}, the base-cell counts:

client->query_precipitation();
// ... draining ...
case SimEventKind::Values: {
    VoxelDims d = event->value_dims();          // {nx, ny, 1}
    Slice<const float> p = event->values();      // mm, row-major nx*ny
    float peak = 0.0f;
    for (size_t i = 0; i < p.size(); ++i) peak = std::max(peak, p[i]);
    std::printf("rain map %u x %u, peak %.1f mm\n", d.x, d.y, peak);
    break;
}

Use it to drive ground wetness, puddles, flooding, or a rainfall overlay.

Saving and restoring: snapshots

You can serialize the entire simulation to bytes and restore it later - for save games, deterministic replays, or rewinding:

  • save_snapshot() answers with a Snapshot event; read the bytes with snapshot_bytes() and store them however you like (copy them out before the event drops).
  • load_snapshot(bytes) rebuilds the sim from those bytes and answers with SimReady (or Error on a malformed buffer).

A snapshot captures the grid, fields, refinement, precipitation, and time - but not the installed scenario or regridder. After loading, reinstall your scenario and regridder if the run still needs them (reseeding is safe - seeds are idempotent, from Scenarios).

Takeaways

  • Grabs sample a dense voxel box for rendering: grab_presets for render-ready Cloud/Rain/Condensate/Temperature, grab_fields for raw fields. Define a ViewBox (resolution independent of the sim grid), take_grab() the result, upload each channel(i) (z-slice major), and always recycle the buffer.
  • Probes return a live SoundingReportData (CAPE, shear, …) for one column - the tool for meteorological gameplay logic.
  • Queries are cheap scalars: query_max (domain peaks), query_precipitation (rain map, mm), query (status). Prefer these over grabs when you only need numbers.
  • Snapshots save/restore the whole sim, but not the scenario or regridder - reinstall those after load_snapshot.
