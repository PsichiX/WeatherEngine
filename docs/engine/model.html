WeatherEngine - Game-ready weather simulations

How the engine is shaped

You now understand the weather. Part III is the machine that simulates it. This first page is the mental model of how you talk to the engine - which matters a lot, because WeatherEngine is built around one firm rule: the simulation must never stall your game.

The problem it solves

A weather step is heavy. Advancing a storm one tick can take milliseconds - sometimes many. If your game called into the simulation and waited for it to finish, every heavy step would freeze the frame. That is unacceptable in a game running at 60 or 120 frames per second.

So WeatherEngine does not run on your game thread at all. It runs on its own background worker thread, and you talk to it by message-passing. You send it a command and immediately get on with your frame; later, you pick up the result whenever it is ready. Nothing you call ever blocks.

The actor pattern

This design is called an actor: one owner of the state, talked to only through messages.

   your game thread                         the worker thread
  ------------------                       -------------------
   client->advance(...)  --- command --->   (parks until a command
   client->grab_...(...) --- command --->    arrives, then runs the
                                             heavy step here)
   client->poll()        <-- event ------   posts results back
   client->poll()        <-- event ------
  • You send commands with methods on a SimClient: create_sim, advance, grab_presets, probe, and so on. Each one is non-blocking - it drops a message in a queue and returns right away.
  • The worker owns the actual simulation. It sleeps when the queue is empty and wakes up to run each command. Heavy work happens here, off your frame.
  • You poll for results with poll(), which hands back one SimEvent at a time, or an empty one if nothing is ready yet. You never wait; you just check what has arrived and move on.

The key property: there is no shared, locked state between the two threads. Data moves only as messages. That is what keeps the game thread free.

Sending a command: the request id

Every command method returns a uint64_t request id:

uint64_t id = client->advance(5, 5);
if (id == 0) {
    // The command queue was full - the worker is behind. Skip this
    // frame's command and try again next frame. This is normal
    // backpressure, not an error.
}
  • A non-zero id means the command was accepted and queued. The event that eventually answers it will carry the same id in event->seq(), so you can match responses to requests if you need to.
  • Zero means the request queue was full. The queue is bounded (it holds a few hundred commands), so this only happens if you fire commands far faster than the worker drains them. The right response is simply to skip and retry next frame - never to spin or block.

Receiving results: poll and drain

Results come back as events. Each frame you drain the event queue - poll repeatedly until it is empty - and act on whatever you find:

for (;;) {
    Box<SimEvent> event = client->poll();
    SimEventKind kind = event->kind();
    if (kind == SimEventKind::None) {
        break;  // queue empty - done for this frame
    }
    switch (kind) {
    case SimEventKind::Advanced:  /* read event->status()   */ break;
    case SimEventKind::Grabbed:   /* take the volume, render */ break;
    case SimEventKind::Regridded: /* read the regrid summary */ break;
    case SimEventKind::Error:     /* log error_message()     */ break;
    default: break;
    }
}

An event is a small tagged object. event->kind() tells you which variant it is; then you call the matching accessor. Here is the full set:

Nothing is ever pushed to you unsolicited. Every event answers a command you sent. If you did not ask for a grab, no grab arrives.

No exceptions, ever

WeatherEngine never throws. A bad configuration - an impossible grid, a physically unstable time step, a malformed snapshot - does not crash or abort. It comes back as an Error event with a human-readable message:

case SimEventKind::Error:
    std::printf("sim error: %s\n",
                std::string(event->error_message()).c_str());
    break;

This means you can experiment freely. The worst a wrong number does is produce an Error you can read and fix.

The threading budget

When you spawn the worker you also set how many CPU cores its heavy math may use:

Box<SimClient> client = new_sim_client(8);   // cap the sim at 8 cores
std::printf("using %u threads\n", client->worker_thread_count());

new_sim_client(worker_threads) builds a private thread pool of exactly that many threads, and all simulation work runs inside it. That is a hard cap: the weather can never grab more than those cores, so a heavy step cannot saturate the machine and starve your renderer.

  • Pass a number a few below your core count - e.g. 8 on a 12-thread CPU - to leave headroom for the game and the graphics engine.
  • Pass 0 to mean “every logical core” (use only if the sim has the machine to itself).
  • worker_thread_count() reports the resolved value (never 0).

The lifecycle at a glance

Everything in the rest of Part III fits this sequence:

  1. Describe the air - author a Sounding (Part II).
  2. Spawn the worker - new_sim_client(threads).
  3. Build the world - create_sim(config, axis, sounding, options), wait for SimReady. See Building a world.
  4. Start the weather - install_scenario(...) (seeds, feeds, air masses), optionally set_regridder(...). See Scenarios and the adaptive grid.
  5. Each frame - advance(...), grab_*(...), then drain poll(). Render the grabbed volumes. See Getting data back out.
  6. Shut down - drop the SimClient (or call shutdown()) to stop and join the worker.

Takeaways

  • The simulation runs on a background worker thread; you talk to it through a SimClient by message-passing. Nothing blocks your frame.
  • Commands are non-blocking and return a request id; 0 means the queue was full - skip and retry, it is normal backpressure.
  • Results are events you poll() and drain each frame; match on kind() and call the right accessor. Nothing arrives unsolicited.
  • No exceptions - failures are Error events you can read.
  • new_sim_client(threads) caps how many cores the weather can use; leave headroom for the game.
